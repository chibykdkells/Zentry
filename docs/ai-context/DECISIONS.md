## ADR-001: Money stored as BigInt in Kobo

**Date:** 2026-04-03
**Status:** ACCEPTED

**Decision:** All monetary values stored as `BigInt` in PostgreSQL using Prisma.
Unit is Kobo (1 Naira = 100 Kobo). Example: ₦1,500.00 = `150000n`.

**Reason:** Floating point arithmetic is inherently imprecise and unsuitable for
financial applications. `0.1 + 0.2 !== 0.3` in IEEE 754. Integer arithmetic
(Kobo) eliminates rounding errors entirely. BigInt is used because Kobo values
for platform-scale transactions can exceed JavaScript's safe integer limit.

**Alternatives considered:**
- `Decimal` type (Prisma) — adds complexity, still risk of display errors
- String storage — loses arithmetic capability, requires parsing everywhere
- Float — rejected immediately due to precision issues in finance

**Consequences:**
- All API responses must format Kobo to Naira string before sending to client
- Shared utility `koboToNaira()` and `formatNaira()` used everywhere
- Frontend never performs money arithmetic — server is source of truth

## ADR-002: UUID v4 for all primary keys

**Date:** 2026-04-03
**Status:** ACCEPTED

**Decision:** All database primary keys use UUID v4 strings generated by
PostgreSQL's `uuid_generate_v4()` via Prisma's `@default(uuid())`.

**Reason:**
- Sequential integers expose record counts (user #1, user #2 reveals platform size)
- Sequential IDs enable enumeration attacks (try IDs 1, 2, 3... to scrape data)
- UUIDs are globally unique — safe for distributed systems and future sharding
- UUIDs don't reveal business data in URLs

**Alternatives considered:**
- Auto-increment integers — rejected (enumeration risk, business data exposure)
- ULID — good option but adds a dependency; UUID is sufficient for current scale
- NanoID — good for short IDs but less standard tooling support

**Consequences:**
- Slightly larger storage footprint (16 bytes vs 4 bytes for int)
- Cannot sort by ID to get insertion order — use `createdAt` for ordering
- URLs contain UUIDs — acceptable for our use case

## ADR-003: Provider Abstraction Layer (PAL)

**Date:** 2026-04-03
**Status:** ACCEPTED

**Decision:** All external provider integrations (payment gateways, VTU APIs,
SMS, email, storage) are abstracted behind TypeScript interfaces. Business
logic only ever calls the interface, never the concrete adapter.

**Reason:**
- Nigerian fintech market has provider instability — gateways go down, APIs change
- Client expressed explicit requirement: switching providers must require minimal code change
- Testability: interfaces can be mocked in unit tests without real API keys
- Separation of concerns: business logic is independent of vendor implementation

**Alternatives considered:**
- Direct integration — rejected (tight coupling, high switching cost)
- Third-party abstraction library — none suitable for Nigerian market specifically
- Microservices per provider — over-engineering for current scale

**Consequences:**
- Every new provider requires an adapter class implementing the interface
- Active provider configured via environment variable
- All adapters registered in the provider module, only active one delegates calls
- Provider-specific features (e.g. FintavaPay-specific fields) not exposed to business logic

## ADR-004: NestJS for backend

**Date:** 2026-04-03
**Status:** ACCEPTED

**Decision:** Backend built with NestJS (Node.js framework, TypeScript-first).

**Reason:**
- Modular architecture enforced by framework — prevents spaghetti code at scale
- Dependency injection built-in — enables testability and PAL pattern
- Decorators align with TypeScript style — `@Injectable`, `@Controller`, `@Guard`
- Strong conventions reduce decision fatigue on a large codebase
- Excellent WebSocket support (Socket.io) built-in
- Bull queue integration well-maintained for NestJS

**Alternatives considered:**
- Express.js — too unopinionated, requires too many architectural decisions
- Fastify — good performance but less ecosystem for our specific needs
- Hono — excellent but newer, less proven at this scale

**Consequences:**
- All modules follow NestJS module pattern
- Dependency injection via constructor only (no service locator)
- Feature flags and config via NestJS ConfigModule

## ADR-005: Next.js App Router for frontend

**Date:** 2026-04-03
**Status:** ACCEPTED

**Decision:** Frontend built with Next.js App Router (currently Next.js 16, not Pages Router).

**Reason:**
- Server Components enable faster initial page load (critical for mobile users in Nigeria)
- App Router route groups (`(auth)`, `(dashboard)`, `(cbt)`, `(admin)`) provide
  clean layout isolation per role
- App Router route groups (`(auth)`, `(dashboard)`, `(cbt)`, `(admin)`) provide
  clean layout isolation per role
- Current framework version is Next.js 16, which fits the same architectural
  decision while updating the implementation to the actively used release line
- Vercel deployment is first-class and handles edge cases automatically
- TypeScript support is excellent

**Alternatives considered:**
- Vite + React SPA — no SSR, worse SEO, no route-level layouts
- Remix — good but less ecosystem maturity for our stack
- Pages Router — legacy, lacks layout nesting capability

**Consequences:**
- Server Components used for static/read-heavy pages
- Client Components (`"use client"`) for interactive components
- Data fetching via TanStack Query in Client Components

## ADR-006: pnpm + Turborepo monorepo

**Date:** 2026-04-03
**Status:** ACCEPTED

**Decision:** Monorepo managed with pnpm workspaces and Turborepo for
build orchestration.

**Reason:**
- Shared packages (`types`, `utils`, `validators`) eliminate duplication between
  frontend and backend
- Zod schemas defined once in `packages/validators` — used by both NestJS DTOs
  and React form validation
- TypeScript types shared — no duplication of interface definitions
- Turborepo caches builds intelligently — faster CI

**Alternatives considered:**
- Separate repos — rejected (types drift between frontend/backend over time)
- Nx — powerful but higher learning curve
- Yarn workspaces — pnpm is faster and has stricter dependency resolution

**Consequences:**
- All shared logic must live in packages, not in apps
- Changes to shared packages trigger rebuilds of dependent apps (Turborepo handles this)
- `pnpm` required for development — npm/yarn not supported in this repo

## ADR-007: Escrow released by Bull queue only

**Date:** 2026-04-03
**Status:** ACCEPTED

**Decision:** Escrow funds are released ONLY by a Bull queue job that fires
2 hours after result upload. No API endpoint can trigger escrow release.

**Reason:**
- Prevents any client-side manipulation of the release timing
- Ensures the dispute window is always honored — no CBT can pressure-release
- Queue jobs are durable — if server restarts, job survives in Redis
- Makes the system auditable — every release has a queue job record

**Alternatives considered:**
- Cron job — less precise timing, harder to cancel per-order
- Client-triggered with server validation — rejected (too many edge cases, abuse risk)
- Database trigger — harder to test, couples DB to business logic

**Consequences:**
- Redis must be running for escrow to function (critical infrastructure)
- Bull queue must be monitored — failed jobs alert the ops team
- If dispute is raised, the Bull job for that order must be explicitly cancelled

## ADR-008: Wallet PIN separate from login password

**Date:** 2026-04-03
**Status:** ACCEPTED

**Decision:** Users set a separate 6-digit numeric PIN for wallet/financial
actions. This is independent of the login password.

**Reason:**
- Two-factor protection for financial operations (know your password AND your PIN)
- Common UX pattern in Nigerian fintech — users expect this (GTBank, Opay, etc.)
- If login session is compromised, attacker still cannot withdraw without PIN
- PIN is shorter and numeric — designed for quick mobile entry

**Alternatives considered:**
- Require password re-entry for financial actions — worse UX, same security
- No second factor — rejected (insufficient security for financial platform)
- TOTP (Google Authenticator) — too complex for target demographic

**Consequences:**
- All withdrawal requests require PIN verification
- PIN hash stored separately in User model (`walletPin` field)
- PIN lockout after 5 wrong attempts (15-minute cooldown)
- "Forgot PIN" requires identity re-verification (not just email — security measure)

## ADR-009: FintavaPay as primary payment gateway

**Date:** 2026-04-03
**Status:** ACCEPTED

**Decision:** FintavaPay is the primary gateway for wallet funding and CBT
withdrawals. Paystack and Flutterwave are registered as backup adapters.

**Reason:** Client's explicit preference. FintavaPay is the preferred provider
for this market. Paystack and Flutterwave added as backup in case of
FintavaPay downtime or failure — switchable via environment variable.

**Alternatives considered:** Only the above three considered per client instruction.

**Consequences:**
- FintavaPay adapter implemented first and fully tested
- Paystack and Flutterwave adapters implemented as backups (may have reduced
  feature parity initially but must implement the full IPaymentProvider interface)
- Webhook endpoints handle all three gateways simultaneously

## ADR-010: Mobile-first PWA, no native app

**Date:** 2026-04-03
**Status:** ACCEPTED

**Decision:** Zentry is a Progressive Web Application only. No native iOS/Android
app is planned for launch.

**Reason:**
- PWA can be installed on both Android and iOS — covers 100% of mobile market
- Single codebase for web + mobile-like experience
- No App Store/Play Store approval delays at launch
- Significantly lower development cost
- Nigerian users increasingly comfortable with PWA installs (WhatsApp Web, etc.)

**Alternatives considered:**
- React Native — separate codebase, longer timeline
- Flutter — separate codebase, different team skills required
- Capacitor (wrap PWA as native) — adds complexity, can be done post-launch

**Consequences:**
- All UI must be designed mobile-first
- Bottom navigation (not sidebar) is primary nav on mobile
- PWA offline support required for poor network conditions (common in Nigeria)
- Push notifications via Web Push API (PWA) — slightly different from native push

## ADR-011: White-label expansion will be a multi-tenant Phase 2/3 track

**Date:** 2026-04-05
**Status:** ACCEPTED

**Decision:** Zentry will support a future white-label offering as a proper
platform-first multi-tenant expansion after the core product is stabilized.
Zentry is the infrastructure/platform layer. The launch business will become a
first-party tenant operating under the same tenant model as every other tenant,
not a privileged special case. Each tenant will have isolated users, staff,
orders, wallet views, disputes, reports, branding, and tenant admin controls.
Tenant users will exist only inside their tenant portal. Initial tenant
launches will use Zentry-managed providers only, with tenant-managed providers
considered later for VTU and NIN services.

**Reason:**
- White-labeling materially expands the business from a direct platform into a
  B2B SaaS platform
- Treating the launch business as a normal tenant produces a cleaner long-term
  architecture and avoids special-case product logic
- Tenant isolation, domain routing, provider resolution, and settlement rules
  affect the database, auth, permissions, billing, and frontend architecture
- Treating this as a cosmetic branding feature would create expensive rework and
  weak data-isolation boundaries
- Deferring it until after core stabilization reduces delivery risk

**Alternatives considered:**
- Add branding only with no tenant isolation — rejected (not enough for the
  business model)
- Build white-label in Phase 1 immediately — rejected (too disruptive while the
  core platform is still stabilizing)
- Allow tenant-owned providers from day one — rejected (too much operational and
  security complexity for the first white-label release)

**Consequences:**
- A dedicated roadmap is required before coding the expansion
- Future schema work must introduce `Tenant` as a first-class concept
- The first live operating business must eventually run through the tenant model
- Auth, routing, provider resolution, and billing will become tenant-aware in a
  later expansion track
- PWA behavior and security requirements must remain first-class constraints in
  the expansion, not cleanup work added later
- Current Phase 1 work must not quietly introduce partial multi-tenancy without
  following the roadmap
