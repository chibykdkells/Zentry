# DATABASE.md — Zentry Database Rules & Schema Guide

> Last updated: 2026-04-03
> Read this before writing any Prisma schema, migration, query, or
> anything that touches money or IDs.

---

## Hard Rules — Never Violate

### Money

```
ALL monetary values are stored as BigInt in KOBO.
1 Naira = 100 Kobo.
₦1,000.00 = 100000 (BigInt)

NEVER use Float, Decimal, or Number for money.
NEVER store "1000.00" as a string.
NEVER do arithmetic on money outside of BigInt operations.
```

Utility functions (packages/utils/src/kobo.ts):
```typescript
nairaToKobo(naira: number): bigint   // 1000 → 100000n
koboToNaira(kobo: bigint): string    // 100000n → "1000.00"
formatNaira(kobo: bigint): string    // 100000n → "₦1,000.00"
```

Always use these. Never inline the conversion.

### IDs

```
ALL primary keys are UUID v4 strings.
@id @default(uuid())

NEVER use @default(autoincrement()) on any model.
NEVER use integer IDs — they expose count data and enable enumeration attacks.
```

Exception: `sortOrder` fields on Services and Categories use Int (not a PK).

### Timestamps

```
ALL timestamps stored as UTC DateTime in PostgreSQL.
@default(now()) for createdAt
@updatedAt for updatedAt
```

Never store timestamps as strings or Unix epoch integers.

## ORM Rules

- **All database access goes through Prisma.** No raw SQL unless:
  1. The operation is genuinely impossible in Prisma (document in DECISIONS.md)
  2. It uses `$queryRaw` with typed parameterized inputs (never string concat)
- Use `prisma.$transaction([])` for any operation that touches multiple tables.
- Wallet balance updates MUST always be inside a transaction.
- Never update `availableBalance` or `escrowBalance` directly — always go
  through the wallet service which logs a Transaction record atomically.

## Schema Design Principles

### Wallet Balance Updates (Critical Pattern)

Every balance change must:
1. Create a `Transaction` record with `balanceBefore` and `balanceAfter` snapshots
2. Update the `Wallet` record
3. Both steps in a single `prisma.$transaction()`

```typescript
// CORRECT
await prisma.$transaction([
  prisma.transaction.create({ data: { balanceBefore, balanceAfter, ... } }),
  prisma.wallet.update({ where: { id }, data: { availableBalance: newBalance } }),
])

// WRONG — never do this
await prisma.wallet.update({ data: { availableBalance: { increment: amount } } })
// (no transaction log = no audit trail = financial liability)
```

### Order Financial Snapshot

When an order is created, snapshot the current service pricing into the order:
```
order.totalAmount    = service.totalPrice (at time of order)
order.platformFee    = service.platformFee (at time of order)
order.cbtCommission  = service.cbtCommission (at time of order)
```

This ensures price changes don't retroactively affect existing orders.

### JSON Fields

Used for dynamic/variable data:
- `Service.requiredDocuments` — array of document descriptors
- `Service.requiredFields` — dynamic form field definitions
- `Order.submittedData` — user's form submission
- `Order.requesterDocUrls` — array of uploaded file URLs
- `Order.providerResponse` — raw VTU API response
- `Transaction.metadata` — gateway webhook payload

Always type JSON fields with TypeScript interfaces in `packages/types`.
Never store untyped blobs.

## Naming Conventions

| Thing | Convention | Example |
|---|---|---|
| Model names | PascalCase singular | `User`, `Order`, `CbtProfile` |
| Field names | camelCase | `createdAt`, `walletId`, `isActive` |
| Enum values | SCREAMING_SNAKE_CASE | `CBT_CENTER`, `ESCROW_LOCK` |
| Index names | auto-generated by Prisma | (don't override) |
| Relation names | descriptive strings | `"OrderRequester"`, `"OrderCbt"` |

---

## Indexes Strategy

Add `@@index` on:
- All foreign key fields (if Prisma doesn't add automatically)
- Fields used in WHERE clauses frequently: `status`, `role`, `isActive`
- `createdAt` on high-volume tables (Transaction, Order, Notification)
- `reference` on Transaction (unique lookups)
- `orderNumber` on Order (human lookups)
- `email`, `phone` on User

Do NOT add indexes speculatively. Only index what is actually queried.

## Migration Rules

- Never edit a migration file after it has been applied to any environment.
- Always run `prisma migrate dev --name <descriptive-name>` (never `--force`).
- Migration names: snake_case, descriptive. Example: `add_cbt_suspension_status`
- Before any migration in production: backup the database.
- Destructive migrations (dropping columns, changing types) require explicit
  data migration step documented in a comment at the top of the migration file.

---

## Soft Delete Policy

Zentry does **not** use soft deletes by default. Exceptions:
- Users: use `isActive: false` (never delete users — financial history)
- Services: use `isActive: false` (never delete — order history references)
- Orders: NEVER deleted — immutable financial record
- Transactions: NEVER deleted — immutable financial record

## Seed Data (prisma/seed.ts)

The seed always creates:
1. Super Admin: `admin@zentry.ng` / `Admin@Zentry2024!`
2. Test tenant: `testbiz`
3. Tenant Admin: `tenant@test.com` / `Test@1234!`
4. Test Individual: `user@test.com` / `Test@1234!`
5. Test CBT (approved): `cbt@test.com` / `Test@1234!`
6. Initial service categories: JAMB, NIMC, NECO, VTU_AIRTIME, VTU_DATA, VTU_CABLE_TV, VTU_ELECTRICITY
7. Sample services per category with realistic pricing
8. System config records

Seed is idempotent — running twice does not create duplicates (use `upsert`).

## Tables That Must Never Be Truncated in Production

- `Transaction` — financial ledger
- `Order` — order history
- `AuditLog` — security audit trail
- `Dispute` — legal record
- `WithdrawalRequest` — payout record

These are append-only in production. No delete operations permitted.
